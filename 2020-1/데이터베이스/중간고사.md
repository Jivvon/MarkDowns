###### 1. 오픈소스를 포함하여 관계형 DBMS 종류와 각각의 특장점을 설명하라

#### MariaDB

<img src="https://images.goodfirms.co/cdn-cgi/image/width=700,quality=100,format=auto/https://goodfirms.s3.amazonaws.com/blog/general/1549000927-5.-mariadb.jpg" alt="MariaDB" style="zoom:25%;" />

 - 특징
    - mysql 소스코드를 같이하므로 사용방법과 구조가 mysql과 동일하며 호환 가능하다.
    - plugins, storage engines 등 다양한 데이터베이스 도구로 구성된 빠르고 강력하며 확장 가능한 데이터베이스 서버를 제공한다.
    - 리눅스 운영체제에서 많이 사용한다.
    - 웹환경에 맞게 설계되어있다.
 - 장점
    - 오픈소스 및 관계형 데이터베이스로 개발된 데이터 엑세스를 위한 SQL 인터페이스를 제공한다.
    - mysql에 비해 가볍고 빠르다.
    - Dynamic column 지원을 통해 NoSQL 유형의 기능을 허용한다.
    - 최근 업데이트 기준으로 모든 DB와 호환이 가능하다.
    - 쿼리 최적화 프로그램을 제공한다.
 - 단점
    - 서브쿼리를 지원하지 않으며 실시간 백업을 지원하지 않는다.

#### DB2 Express-c

<img src="https://storage.ning.com/topology/rest/1.0/file/get/294191597?profile=RESIZE_180x180&amp;crop=1%3A1&amp;width=82" alt="DB2 Express-C Community – ChannelDB2" style="zoom:90%;" />

- 특징
  - JAVA, C, C++ 과 같은 프로그래밍 언어를 사용하여 애플리케이션을 구현할 수 있다.
  - 설정이 빠르고 사용이 쉬워 학계, 기업 등 많은 환경에 적합하다.

- 장점
  - 매우 빠르고 안전하며 안정적이고 확장성이 뛰어나다.
  - 데이터베이스의 크기와 사용자 수의 제한이 없다.
  - 데이터는 전송하거나 보관할 때에 항상 암호화되어 보안이 뛰어나다. 심지어 row, col의 엑세스 제어가 가능하다.
  - 다수의 사용자가 서로 관계된 데이터베이스에 동시에 접근할 수 있다.

#### SQLite

<img src="https://images.goodfirms.co/cdn-cgi/image/width=700,quality=100,format=auto/https://goodfirms.s3.amazonaws.com/blog/general/1549000905-3.-sqlite.jpg" alt="SQLite" style="zoom:25%;" />

- 특징 
  - 세계적으로 가장 널리 배포된 데이터베이스로, 많은 프로젝트가 포함된
  - 로컬에서 사용하는 경량 데이터베이스이다. 
  - 아이폰이나 안드로이드 등의 모바일에서 많이 활용된다.
  - 트랜잭션을 제공하는 파일 기반의 DMBS이며 잦은 파일 접근을 최소화할 수 있다.

- 장점 
   - 윈도우, 맥OS, 리눅스, iOS, 안드로이드를 비롯하여 대부분의 OS에서 실행 가능하다
   - 다양한 서드파티 ORM 및 데이터 계층이 있어 원시 SQL 문자열을 사용하지 않아도 된다.
   - 원격 Connection이 발생하지 않기 때문에 데이터로의 접근이 매우 빠르다. 
   - 트랜잭션과 원자성 동작을 지원하여 프로그램이 충돌하여도 데이터베이스가 손상되지 않는다.
   - 개인 소유가 아니라 자유롭게 코드를 사용할 수 있다.

#### CUBRID

<img src="https://images.goodfirms.co/cdn-cgi/image/width=700,quality=100,format=auto/https://goodfirms.s3.amazonaws.com/blog/general/1549000896-2.-cubrid.jpg" alt="SQLite" style="zoom:25%;" />

- 특징
  - 데이터 공유, 고성능 데이터 캐싱, 객체 지향 데이터베이스 요소 관계, 기본 미들웨어 브로커, 확장 가능하고 사용자정의 가능한 글로벌화 지원

- 장점
  - MySQL 및 기타 알려진 데이터베이스와 함께 높은 수준의 SQL 호환성 제공한다.
  - 강력한 오픈 소스 소프트웨어를 제공하며 안정적이고 높은 성능을 제공한다.
  - 사용자가 웹 사이트를 위한 맞춤형 데이터베이스 솔루션을 구축할 수 있는 플랫폼을 제공한다.
  - 테이블, 데이터베이스 및 인덱스의 수와 크기를 무제한으로 제공한다.
  - 다중 서버 아키텍처, 다중 스레드, 웹 서비스를 위한 기본 브로커 미들웨어 기능을 제공한다.

<img src="https://images.goodfirms.co/cdn-cgi/image/width=700,quality=100,format=auto/https://goodfirms.s3.amazonaws.com/blog/general/1549000937-6.-firebird.jpg" alt="FirebirdSQL" style="zoom:25%;" />

  - 설치크기가 수MB에 불과할 정도로 작고 빠르며, 메모리 또한 수십MB 정도로 적게 소모한다.
  - 동적으로 sql client 애플리케이션 개발을 지원한다.
  - 다수의 클라이언트에게 단일 데이터베이스를 제공함으로써 자원 이용성이 높다
  - 다수의 데이터베이스에 대한 보안이 좋다.(충돌이 잘 일어나지 않음)
  - 서버엔진에 대한 부담이 적어 개발에 용이하다.

###### 2. SQL의 이론적 배경이 된 Relational Algebra 및 Relational Calculus에 대해서 각각 설명하고 차이점을 설명하라

### Relational Algebra 관계 대수

원하는 정보를 **어떻게(How) 유도하는가**를 연산자와 연산규칙을 이용하여 기술하는 **절차적 언어**이며, 피연산자와 결과가 모두 릴레이션이다.

- 산술 연산자 (일반 집합 연산자)

  집합 연산자의 입력으로 사용되는 두 개의 릴레이션은 합집합 호환이어야 한다.

  - 합집합 (R ∪ S)
  - 교집합 (R ∩ S)
  - 차집합 (R - S) : 릴레이션 R에는 속하지만 릴레이션 S에는 속하지 않는 튜플로 이루어진 릴레이션 생성
  - Cartesian Product (카테시안 곱) (R x S) : R의 모든 튜플과 S의 모든 튜플을 결합 하여 릴레이션 생성

- 관계 연산자
  - Selection ($\sigma$) : 릴레이션에서 조건에 해당하는 **Tuple**을 추출하는 것. (Row)
  - Projection ($\Pi$) : 릴레이션에서 특정 **Attributes**를 추출하는 것. (Column)
  - Join ($\Join$) : 두 개의 릴레이션을 연결하여 하나의 릴레이션이 생성된 것과  같은 역할 수행. (여러 릴레이션을 합쳐 원하는 Column만을 추출)
    - 동등 조인 (equijoin) : 중복된 Column 허용.
    - 자연 조인 (natural join) : 중복된 Column 허용하지 않음. (일반적인 Join)
    - 세타 조인 (theta join)
    - 외부 조인 (outer join)
      - 왼쪽 외부 조인 (Left Join) : 왼쪽의 릴레이션은 전부 포함. 오른쪽 릴레이션은 대응되는 튜플이 없을 경우 NULL
      - 오른쪽 외부 조인 (Right Join) : 오른쪽의 릴레이션은 전부 포함. 왼쪽 릴레이션은 대응되는 튜플이 없을 경우 NULL
      - 완전 외부 조인 : 다른 릴레이션에 대응되는 튜플이 없거나, NULL일 경우에도 조인 결과에 포함한다.
    - 세미 조인 (semijoin)
  - Division ($\div$) : 한 릴레이션에서 다른 릴레이션의 Attribute를 제외한 **나머지 Attribute만을 추출**하는 것.
  - Rename ($\rho$) : 릴레이션의 이름 변경.
  
- 기타 연산자

  - 집계 함수 : 여러 개의 튜플을 입력받아 하나의 결과를 낸다
    - SUM, AVERAGE, MAXIMUN, MINIMUM, COUNT

> **Q.** 부품 릴레이션(P)에서 부품보관장소(CITY)가 ‘서울’이고 부품색이 ‘빨강’인 튜플들을 삭제하여 보라
> **A.** (부품)-(σ부품보관장소=‘서울’∧부품색=‘빨강’(부품))

> **Q.** 납품업자 릴레이션(S)에서 모든 납품업자의 납품번호(SNO)를 추출하여 보라.
> **A.** $\Pi$납품번호(납품업자)

> **Q.** 납품정보 릴레이션(SP)에서 납품수량이 300보다 큰 납품업자의 납품번호(SNO)와 부품번호(PNO)를 추출하여 보라.
> **A.** $\Pi$납품번호,부품번호(σ납품수량>300(납품정보))

> **Q.** 납품정보 릴레이션(SP)의 납품수량(QTY)이 300보다 큰 납품업자의 납품번호(SNO)와 이름(SNAME), 그리고 소재지(CITY)를 추출하여 보라
> **A.** $\Pi$납품번호,이름,소재지(σ납품수량>300(납품업자⋈납품번호=납품번호납품정보))

`+ 참고`

> Selection 결과 중복 튜플이 존재할 수 없지만, Projection 결과에서는 중복 튜플이 존재할 수 있다.
>
> RDB에서 대부분의 질의는 Selection, Projection, Natural Join으로 표현 가능하다.

##### 관계 대수의 완전성

>**Selection, Projection, 합집합, 차집합, Cartesian Product**는 관계 대수의 필수적인 연산자이다. 다른 관계 연산자들은 이들을 조합하여 표현 가능하며, 임의의 질의어가 적어도 필수적인 관계 대수 연산자들만큼의 표현력을 갖고 있으면 **관계적으로 완전(relationally complete)하다**고 말한다.

### Relational Calculus 관계 해석

수학의 술어 해석에 기반을두고 관계 데이터베이스를 위해 제안한 **질의어로 표현하는 비절차적 언어**이다. 원하는 정보가 무엇(What)이라는 것만 정의하며, 이를 정의할 때 계산수식을 사용한다.

- 튜플 관계 해석 (Tuple Relational Calculus)

  - 튜플 변수 : 릴레이션의 튜플들이 범위로 가지는 변수.

  - {t1.A1, t2.A2, ..., tn.An | COND(t1, t2, ..., tn, tn+1, tn+2, ..., tn+m)}

    - t1,t2,...,tn,tn+1,tn+2,...,tn+m은투플변수
    - 각 Ai는 ti가 범위로 하는 릴레이션의 Attribute
    - COND는 조건 또는 투플 관계 해석의 식

    식을 구성하는 각 원자는 특정한 투플들의 조합에 대해서 참(true) 또는 거짓(false)으로 계산되며, 계산된 결과값을 원자의 진리값이라 부른다.

  - 관계 대수의 UNION은 관계 해석의 or 연결자에, INTERSECTION은 and 연결자에 대응한다.

  - 
    not 연결자는 전체 정량자(∀)와 존재 정량자(∃)를 동등한 식으로 변환하는 데에 사용될 수 있다.

- 도메인 관계 해석 (Domain Relational Calculus)

  - 도메인 변수 : 한 Attribute의 도메인을 범위로 가지는 변수.
  - 차수가 n인 릴레이션의 경우 n개의 도메인 변수를 사용한다.

###### 예시

> **Q.** Research 부서에서 일하는 모든 사원의 이름과 주소를 검색하라.
>
> **A.** {t.FNAME, t.LNAME, t.ADDRESS | EMPLOYEE(t) and (∃d) (DEPARTMENT(d) and d.DNAME = ‘Research’ and d.DNUMBER = t.DNO)}

> **Q.** 각 사원에 대하여 사원의 이름과 성, 직속 상사의 이름과 성을 검색하라.
>  **A.** {e.FNAME, e.LNAME, s.FNAME, s.LNAME | EMPLOYEE(e) and EMPLOYEE(s) and e.SUPERSSN = s.SSN}

#### 관계 대수와 관계 해석의 차이점

**관계 대수**는 관계형 데이터베이스의 릴레이션을 조작하기 위하여 **데이터 획득 절차를 명세해주는 절차 언어**이다. 즉, 절차 언어는 원하는 **데이터를 얻기 위하여 어떻게(HOW)해야 되는지** 일련의 연산을 순서적으로 명세해주어야 한다. 

**관계 해석**은 관계형 데이터베이스 릴레이션을 조작하기 위하여 **데이터 획득 절차를 명세해 주지 않는 비절차적 언어**이다. 즉, 비절차적 언어는 **원하는 정보가 무엇(WHAT)이라는 것**만 선언 하면 된다.

관계해석과 관계대수는 관계 데이터베이스를 처리하는 기능과 능력면에서 모두 **동등**하며 **관계대수로 표현한 식은 관계해석으로도 표현할 수 있다**.
